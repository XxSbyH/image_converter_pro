## Priority 0.1: 修复删除功能逻辑错误（最高优先级）

### 问题描述

**当前行为**：
- 失败的文件无法删除（截图 3）
- 待处理的文件无法删除（截图 4）
- 系统提示"处理中无法删除文件"

**错误原因分析**：
- 删除逻辑判断错误
- 只有"处理中"状态才应该禁止删除
- 其他所有状态（待处理、已完成、失败）都应该可以删除

### 改进目标

**修复删除逻辑规则**：

| 文件状态 | 是否可删除 | 说明 |
|---------|----------|------|
| 待处理 (pending) | ✅ 可以删除 | 还未开始处理，随时可删 |
| 处理中 (processing) | ❌ 不可删除 | 正在转换，禁止删除 |
| 已完成 (completed) | ✅ 可以删除 | 已完成，可以删 |
| 失败 (failed) | ✅ 可以删除 | 失败了，更应该删 |

**实施步骤**：

1. **检查删除按钮的点击事件逻辑**
   
   - 定位删除按钮的 onPressed 回调
   - 检查是否有状态判断
   
2. **修改状态判断条件**
   - 当前判断：可能是 `if (_isProcessing) 提示无法删除`
   - 正确判断：`if (file.status == 'processing') 提示无法删除`
   - 其他状态直接执行删除

3. **删除逻辑流程**
   ```
   用户点击删除按钮
   ↓
   检查该文件的状态
   ↓
   如果状态是 'processing'
     → 显示提示："该文件正在处理中，无法删除"
     → 不执行删除
   ↓
   如果状态是其他（pending/completed/failed）
     → 从文件列表中移除该文件
     → 更新 UI
     → （可选）显示提示："已删除"
   ```

4. **处理边界情况**
   - 如果全部文件都在处理中，批量删除应该提示"存在处理中的文件，无法删除"
   - 如果部分文件在处理中，批量删除应该只删除非处理中的文件

### 验收标准

**必须通过的测试**：

```
测试 1: 删除待处理文件
1. 添加 3 个文件到列表（状态：待处理）
2. 点击第 2 个文件的删除按钮
3. ✅ 期望：文件立即从列表移除
4. ❌ 不应该：显示"处理中无法删除"

测试 2: 删除失败文件
1. 开始转换，等待某个文件失败
2. 点击失败文件的删除按钮
3. ✅ 期望：文件立即从列表移除
4. ❌ 不应该：显示"处理中无法删除"

测试 3: 删除处理中文件（正确的禁止场景）
1. 开始转换
2. 趁某个文件还在处理中时点击其删除按钮
3. ✅ 期望：显示提示"该文件正在处理中，无法删除"
4. ✅ 期望：文件仍在列表中

测试 4: 删除已完成文件
1. 转换完成后
2. 点击已完成文件的删除按钮
3. ✅ 期望：文件从列表移除

测试 5: 批量删除（混合状态）
1. 列表中有：2 个待处理、1 个处理中、2 个已完成
2. 点击"清空全部"
3. ✅ 期望：提示"存在正在处理的文件，是否删除其他文件？"
4. ✅ 确认后：删除除了处理中之外的所有文件
```

**预计工作量**：0.5 - 1 小时

## Priority 0.2: 错误信息用户友好化（紧急）

### 问题描述

**当前错误信息**（截图 2）：
```
处理失败: 格式转换失败: DioException [receive timeout]: 
The request took longer than 0:00:30.000000 to receive data. 
It was aborted. To get rid of this exception, try raising 
the RequestOptions.receiveTimeout above the duration of 
0:00:30.000000 or improve the response time of the server.
```

**问题分析**：
1. 错误信息是英文，中文用户看不懂
2. 错误信息是技术术语（DioException、RequestOptions.receiveTimeout）
3. 错误信息太长，占用大量空间
4. 错误信息没有告诉用户该怎么办

### 改进目标

**建立错误信息翻译和简化机制**

#### 步骤 1: 识别常见错误类型

**网络/超时类错误**：
- `DioException [receive timeout]` → 超时错误
- `DioException [connect timeout]` → 连接超时
- `DioException [send timeout]` → 发送超时
- `SocketException` → 网络连接失败
- `Connection refused` → 后端服务未启动

**文件类错误**：
- `FileSystemException` → 文件访问失败
- `Permission denied` → 权限不足
- `No such file or directory` → 文件不存在
- `File too large` → 文件过大

**格式/处理类错误**：
- `Unsupported image format` → 不支持的格式
- `Image decode error` → 图片损坏
- `PIL.UnidentifiedImageError` → 无法识别的图片

**后端返回的错误**：
- HTTP 400 → 请求参数错误
- HTTP 413 → 文件过大
- HTTP 500 → 服务器内部错误
- HTTP 503 → 服务不可用

#### 步骤 2: 创建错误信息映射表

**映射规则**：

| 原始错误关键词 | 用户友好提示 | 建议操作 |
|--------------|------------|---------|
| `receive timeout` | 处理超时 | 文件可能过大，建议降低质量或分批处理 |
| `connect timeout` | 连接超时 | 请检查网络连接或重启应用 |
| `Connection refused` | 服务未启动 | 请重启应用，如问题持续请联系支持 |
| `FileSystemException` | 文件访问失败 | 请检查文件是否被占用或已移动 |
| `File too large` | 文件过大 | 该文件超过 50MB，请选择较小的文件 |
| `Unsupported image format` | 格式不支持 | 请选择 JPG、PNG、WebP 或 HEIC 格式 |
| `Image decode error` | 图片损坏 | 该图片文件可能已损坏，请重新导出 |
| HTTP 413 | 文件过大 | 该文件超过服务器限制，请选择较小的文件 |
| HTTP 500 | 服务器错误 | 服务处理异常，请稍后重试 |

#### 步骤 3: 实现错误信息转换逻辑

**转换流程**：
```
1. 捕获原始错误（英文技术信息）
   ↓
2. 提取错误关键词
   - 使用正则表达式或字符串匹配
   - 例如：检查是否包含 "timeout"、"refused" 等
   ↓
3. 查找映射表
   - 找到对应的用户友好提示
   ↓
4. 组合最终错误消息
   格式：[用户友好提示] - [建议操作]
   例如："处理超时 - 文件可能过大，建议降低质量或分批处理"
   ↓
5. 显示在 UI 上
   - 文件项下方显示简短版本
   - 可选：点击查看详情（显示原始错误）
```

#### 步骤 4: 优化错误显示样式

**当前问题**：
- 错误信息占用 3 行空间
- 红色文字和正常文字混在一起

**改进方案**：

**方案 A：简洁版（推荐）**
```
文件名：雨画晴天1.png
大小：6.90 MB
状态：❌ 处理超时 - 建议降低质量重试
      [重试] [删除]
```

**方案 B：带详情版**
```

大小：6.90 MB
状态：❌ 处理超时
建议：文件可能过大，建议降低质量或分批处理
      [查看详情] [重试] [删除]

点击"查看详情"后弹出对话框：
┌──────────────────────────────────────┐
│ 错误详情                              │
│                                      │
│ 文件：雨画晴天1.png                   │
│ 错误：处理超时                        │
│ 建议：降低质量或减小文件尺寸后重试      │
│                                      │
│ 技术信息（仅供参考）：                 │
│ DioException [receive timeout]:      │
│ The request took longer than 30s...  │
│                                      │
│ [复制错误信息] [关闭]                  │

### 验收标准

**必须通过的测试**：

```

1. 上传一个超大文件（触发超时）
2. ✅ 显示："处理超时 - 文件可能过大，建议降低质量或分批处理"
